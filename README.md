## 1. Суть проекта: Uptime Monitor

**Распределенная система мониторинга доступности ресурсов.**
Пользователь через REST API регистрирует URL и задает интервал проверки (например, каждые 30 секунд). Система в фоновом режиме выполняет проверки. Если ресурс возвращает ошибку или время ответа превышает заданный лимит, система генерирует событие «Инцидент», сохраняет его в БД и уведомляет пользователя.

## 2. Обязательные пункты (Монолитная архитектура)

Это база, на которой строится всё остальное. Здесь мы используем стандартные подходы Go.

* **API Layer (chi/gin):**
* Реализация REST: `POST /auth/register`, `POST /auth/login`, `GET/POST/DELETE /targets` (ваши сайты).
* **JWT Middleware:** Проверка токена в каждом запросе. Извлечение `user_id` и сохранение его в `context.Context` для использования в обработчиках.


* **Context (Обязательно):**
* Проброс `ctx` от HTTP-хендлера до репозитория базы данных.
* Использование `context.WithTimeout` для HTTP-клиента воркеров (например, если сайт не ответил за 5 секунд — это статус "Down").


* **Логирование (Structured Logging):**
* Использование любых логов, например `uber-go/zap`.
* Логи должны быть в формате **JSON** (для удобного чтения машинами в будущем).
* Логирование каждого этапа: "задача получена", "запрос отправлен", "ошибка БД", "сервис остановлен".
* Обязательно уровни логирования настроить


* **Worker Pool:**
* **Dispatcher:** Читает цели из БД и отправляет их в канал задач.
* **Workers:** Пул горутин, которые делают `http.Get(url)` с использованием `context`.


* **Database (Postgres):**
* Таблица `users`, `targets` (сайты), `check_logs` (результаты каждой проверки). (ты можешь поменять структуру бд или вообще чего угодно, это просто ориентир)


* **Graceful Shutdown:**
* Перехват сигналов `os.Interrupt` и `syscall.SIGTERM`.
* Остановка приема новых задач, ожидание завершения текущих воркеров (через `sync.WaitGroup`) и корректное закрытие соединений с БД.


* **Тестирование:**
* **Testify:** Для читаемых ассертов.
* **Mockery:** Создание моков для интерфейса репозитория. Тестируй логику: если сайт вернул 404, воркер должен правильно записать статус в лог.

## 3. Дополнительные пункты (Микросервисы и масштабирование)

* **Разделение на сервисы:**
1. **Auth/Manager Service:** Работа с пользователями и списком сайтов.
2. **Checker Service:** "Глупый" исполнитель, который умеет только пинговать и слать отчеты.
   (также вместо этого разделения можно написать микросервис отправки email)


* **Kafka (надежное общение между сервисами):**
* API Service пишет сообщение в топик `check_tasks`.
* Checker Service читает из топика, проверяет сайт и пишет результат в топик `check_results`.


* **Redis:**
* Кэширование (например, чтобы отправить письмо только при *первом* падении, а не спамить каждую минуту, пока он лежит).


* **gRPC (быстрое общение между сервисами):**
* Если API сервису нужно мгновенно узнать текущий статус у Checker-а, он делает **gRPC call**. Опиши контракт в `.proto` файле.


* **Docker Compose:**
* Полная инфраструктура: `app-api`, `app-checker` ИЛИ `email-seder`, `postgres`, `kafka`, `redis`
